# 變革管理計畫(Change Management Plan)
## 銀行內部開發者平台(IDP)導入 — 文化融合與組織變革

> **文件用途**:本計畫處理 IDP 技術落地之外,最關鍵也最易失敗的一環——**文化與組織阻力**。
> 技術三個月可成,文化阻力可讓它三年推不動。本計畫提供專業、可執行的變革手段。
>
> **核心方法論**:Kotter 八步變革模型 + 利害關係人分析。
> **配套文件**:`PROJECT_PLAN.md`(技術落地)、`COMPLIANCE_MAP.md`(治理對齊)、`GOVERNANCE_BRIEF.md`(提案)。
>
> **版本**:v1.0 | **最後更新**:2026-06

---

## 0. 給使用者的核心判斷(先讀這段)

在動手前,先內化三個現實:

1. **你面對的反對,多數不是理性的。**
   ISO 對照表能解決「這合不合規」的理性反對。但真正的阻力來自非理性層面:
   怕失去權力、怕被不信任、怕失去掌控、怕擔責任。這些用技術論述打不動。

2. **以你目前的位階,不要試圖「改變整個銀行文化」。**
   那會撞牆,並耗盡你的信用。正確做法是:**做出一個無可辯駁的成功小樣板,
   讓有權力的人「想要」把它擴大。** 文化變革必須由上而下才推得動;
   你的工作是給「那個能由上而下的人」一個無法拒絕的理由。

3. **沒有 Sponsor,不要硬推。**
   找到一個有影響力的主管當靠山,是本計畫成敗的單一最關鍵因素。
   在拿到 Sponsor 之前,你做的是「準備」,不是「推動」。

## 1. 變革願景(一句話)

> **「在不犧牲合規的前提下,讓交付更快。」**

- 對 **主管**:強調「更快」——交付速度、瓶頸消除。
- 對 **稽核 / 資安**:強調「不犧牲合規」——更可追溯、100% 覆蓋。
- 對 **工程師**:強調「少做雜事」——降低認知負荷,專注有價值的工作。

同一個願景,對不同對象用不同的語言切入。這是貫穿全計畫的溝通原則。

## 2. 利害關係人分析(本計畫核心)

### 2.1 利害關係人權力 / 立場矩陣

依「影響力高低」與「對變革的態度」分類,決定投入策略。

| 象限 | 策略 |
|------|------|
| 高影響力 + 支持 | **緊密合作**——納入 Guiding Coalition,讓他們當 Sponsor / 代言人 |
| 高影響力 + 反對 | **重點化解**——優先處理,轉化或至少中立化(最關鍵) |
| 低影響力 + 支持 | **培養為早期採用者**——燈塔專案的種子 |
| 低影響力 + 反對 | **持續溝通**——不耗過多資源,但別忽視其雜音 |

### 2.2 各利害關係人深度剖析

下表是本計畫最重要的部分。對每個角色,拆解:**表面反對什麼 / 真正怕什麼 / 怎麼化解**。

---

#### A. 中階主管(最大阻力源)

- **表面反對**:「自動化審核風險太高」「我們一直都是人工把關」。
- **真正怕的**:審核權 = 權力與存在感。自動化看起來在拿走「我蓋章才能過」的權力,
  讓他們覺得被架空、價值被削弱。
- **化解手段(關鍵:重新定義價值,而非取代)**:
  - 話術從「自動化取代你的審核」改為
    **「自動化幫你擋掉 90% 的瑣碎審核,讓你專注在真正需要判斷的 10% 高風險案子」**。
  - 讓他們成為**護欄規則的定義者**——規則由他們的專業來訂。
    他們從「逐案蓋章的人」升級為「定義安全邊界的人」。權力不是被拿走,是被升維。
  - 在燈塔專案中,公開讓某位中階主管掛名為「規則的把關者」,給予功勞與能見度。

---

#### B. 資安部門

- **表面反對**:「看不懂的自動化等於失去掌控」「誰保證這些規則沒漏洞」。
- **真正怕的**:失去對安全的掌控感;擔心自動化是個無法檢視的黑盒,出事仍是他們扛。
- **化解手段**:
  - 強調**護欄規則由資安共同定義、可檢視、有版控、有測試**——不是黑盒。
  - 用 `COMPLIANCE_MAP.md` 證明每條規則對應到 ISO 控制項。
  - 把資安拉進 Guiding Coalition,讓他們是「規則的共同作者」而非「被通知的一方」。
  - 主打對他們最有吸引力的點:**從抽樣稽核變 100% 覆蓋,他們的掌控其實是增強,不是削弱。**

---

#### C. 稽核 / 內控

- **表面反對**:「Git log 不是我們認的證據格式」「這套怎麼稽核」。
- **真正怕的**:既有的稽核方法失效;看不懂技術紀錄;擔心無法向監管交代。
- **化解手段**:
  - 提供 `TASK-07` 的**稽核證據翻譯層**——把技術紀錄轉成他們看得懂、對應 ISO 編號的報告。
  - 強調人工審核的弱點(抽樣、會累、會放水),自動化反而消除這些弱點。
  - 邀請稽核**共同定義「什麼樣的報告格式他們認可」**,讓他們參與設計而非事後接受。

---

#### D. 第一線工程師(理應是最大受益者,但未必領情)

- **表面反對**:「又一個新工具」「被護欄管很煩」「我自己做得好好的」。
- **真正怕的**:被護欄限制 = 不被信任;學習成本;懷疑這又是上面的形式主義。
- **化解手段(用「平台即產品」思維)**:
  - 讓護欄**盡量無感**——好的護欄是平常感覺不到,只有撞線才知道。
  - **不強迫採用,靠吸引**:讓早期使用者體驗到「省事」,自己當口碑。被命令的會抵抗,自己選的會擁護。
  - 蒐集回饋、快速迭代、公開感謝早期使用者,把他們變成代言人。
  - 強調「護欄幫你扛掉合規責任,出事不是你個人背鍋」——這對工程師其實是減壓。

---

#### E. 高階主管 / 潛在 Sponsor

- **立場**:通常不反對,但需要被「點燃」並給予安全感。
- **他們在乎的**:可量化的商業價值(交付速度、風險下降、成本)、以及「不會出事砸我招牌」。
- **化解 / 爭取手段**:
  - 用**急迫感**打動:目前交付慢 / 憑證外洩風險 / 稽核成本高,再不變的代價。
  - 用燈塔專案的**真實數據**證明可行且低風險。
  - 給他們安全感:這是非生產環境 PoC、用測試資料、有完整治理對齊。
  - **明確請求**:請他們當 Sponsor,提供政治掩護與資源。沒有這層,變革推不動。

## 3. Kotter 八步 × 本專案的具體行動

### 步驟 1:建立急迫感(Create Urgency)
- **做什麼**:用組織在乎的語言陳述痛點,不是「PE 很酷」,而是
  「交付一個環境要等兩週」「憑證外洩風險」「稽核要耗多少人力」。
- **產出**:一頁急迫感簡報,數據化現況痛點。
- **對應**:`GOVERNANCE_BRIEF.md` 的「我們在解什麼問題」。

### 步驟 2:組建領導聯盟(Build Guiding Coalition)
- **做什麼**:爭取一位高階 Sponsor + 資安/稽核的開明派 + 一位願配合的中階主管。
- **關鍵**:這是成敗單一最重要因素。聯盟要橫跨技術、資安、業務,才有正當性。
- **產出**:一份「聯盟名單與各自角色」(誰提供掩護、誰共同定規則、誰當燈塔)。

### 步驟 3:形塑願景與策略(Form a Vision)
- **做什麼**:確立「在不犧牲合規的前提下,讓交付更快」為核心願景。
- **產出**:願景一句話 + 三種對象的差異化版本(見第 1 節)。

### 步驟 4:溝通願景(Communicate the Vision)
- **做什麼**:反覆、多管道、對不同對象用不同語言溝通。
- **原則**:對主管講效率,對稽核講可追溯,對工程師講少做雜事。
- **產出**:利害關係人溝通腳本(可另開一份文件)。

### 步驟 5:授權行動、移除障礙(Empower Action)
- **做什麼**:識別卡關的流程與人,由 Sponsor 出面排除;給燈塔團隊足夠自主權。
- **產出**:障礙清單與排除責任人。

### 步驟 6:創造短期戰果(Generate Short-term Wins)— **你的王牌**
- **做什麼**:執行燈塔專案,在 1-3 個月內做出可量化成果。
- **指標範例**:環境交付時間下降 X%、合規檢查 100% 通過、憑證外洩 0 次、開發者滿意度。
- **原則**:讓數據說話。銀行最信數據和先例。一個成功案例 > 一百次道理。
- **產出**:燈塔專案成果報告(數據化、可對外複述)。

### 步驟 7:鞏固成果、擴大(Consolidate Gains)
- **做什麼**:用第一個戰果的信用,爭取資源擴大到第二、第三個團隊。
- **警告**:不要過早宣告勝利。銀行文化慣性強,鞏固期要夠長。

### 步驟 8:融入文化(Anchor in Culture)
- **做什麼**:讓「自助 + 護欄」變成「這裡本來就這樣做」的預設。
- **手段**:納入新人 onboarding、納入正式流程文件、讓護欄成為標準變更的官方一環。

## 4. 燈塔專案(Lighthouse)選擇準則

第一個示範專案,直接決定整場變革的成敗。選擇標準:

| 準則 | 說明 |
|------|------|
| 痛點明確 | 該團隊的「慢」或「痛」是公認的,改善後對比鮮明 |
| 團隊願配合 | 找「願意嘗試」的團隊,不要選最抗拒的硬啃 |
| 風險可控 | 非生產、非核心系統、用測試資料 |
| 成果可量化 | 能拿出交付時間、合規通過率等硬數據 |
| 能見度適中 | 夠顯眼讓人看到,但不至於失敗就重傷 |

> 避免:一開始就挑最關鍵、最高風險、最多人反對的系統。
> 那不是勇敢,是自殺。先贏一場小的、漂亮的。

## 5. 風險與緩解(變革層面)

| 風險 | 緩解措施 |
|------|----------|
| 找不到 Sponsor | 在拿到 Sponsor 前只做準備,不公開推動;先用小成果吸引潛在 Sponsor |
| 中階主管聯合抵制 | 重新定義其價值(升維為規則定義者),給功勞與能見度 |
| 工程師不採用 | 「平台即產品」,靠吸引非命令;護欄無感化;公開表揚早期使用者 |
| 稽核不認可證據 | 邀稽核共同定義報告格式;提供 ISO 對應翻譯層 |
| 過早宣告勝利 | 延長鞏固期;用第二、三個案例累積,才談制度化 |
| 你個人信用耗盡 | 不越級硬推;聚焦可控範圍做樣板;讓上層去做擴散 |
| 變革被當成「IT 的事」 | 從一開始就把資安、稽核、業務拉進聯盟,變成「大家的事」 |

## 6. 你的下一步(務實行動順序)

1. **盤點利害關係人**:用 2.1 矩陣,把你組織裡的具體人名填進去。
2. **鎖定潛在 Sponsor**:找那個「高影響力 + 可能支持」的人。
3. **準備急迫感材料**:把現況痛點數據化(交付時間、稽核成本、風險事件)。
4. **選定燈塔專案**:用第 4 節準則,挑一個願配合、痛點明確的小團隊。
5. **私下溝通,而非公開宣告**:先一對一爭取關鍵盟友,聯盟成形再對外。

> 記住順序:**先有盟友,再有樣板,再談擴大,最後才是文化。**
> 反過來做(先喊文化變革)幾乎注定失敗。
