# 合規對照表:PE 技術控制 ↔ ISO 27001 / ISO 20000

> 本表是本計畫對銀行的核心價值:把「技術控制」翻譯成「框架控制項」。
> 純技術人做不到,純合規人也做不到——兩邊都懂的人才能做。
> 提案、稽核、內控溝通時,直接引用本表。
>
> 註:ISO 27001 控制項編號採 2022 版 Annex A。實際導入時請以貴行採用的版本與
> SoA(適用性聲明)為準,本表為對應範例,非法遵保證。

---

## 一、核心論述

| 面向 | 人工審核(傳統) | 自動化護欄(本計畫) |
|------|------------------|----------------------|
| 覆蓋率 | 抽樣 | 100% 全面 |
| 時機 | 事後 | 即時 |
| 一致性 | 因人而異、會累、會放水 | 規則固定、不疲勞、不講人情 |
| 可追溯 | 工單 / 簽核文件 | Git 歷史,每行可溯源 |
| 稽核負擔 | 高(需人工彙整) | 低(自動產出證據) |

**結論**:自動化護欄從稽核角度是「降低風險」,不是增加風險。

### Phase 2 — IaC(基礎設施即程式碼)合規護欄

把 Phase 1 的「護欄」概念下沉到基礎設施層:合規寫進可重用模組的預設值,
並以 checkov 在 PR 階段自動把關。控制對應如下:

| 本計畫的技術控制 | 對應任務 | ISO 27001:2022 控制項 | 說明 |
|------------------|----------|----------------------|------|
| secure-bucket 加密 at rest | P2-2 | A.8.24 加密技術使用 | 模組強制 SSE(AES256/KMS),呼叫者無法關閉 |
| 全面封鎖公開存取 | P2-2 | A.8.12 資料外洩防護 / A.5.15 存取控制 | 四項 public-access-block 全 true,焊死 |
| 版本控制(防誤刪/竄改) | P2-2 | A.8.13 資訊備份 | 預設啟用,維護完整性 |
| 強制資源標籤 | P2-2 | A.5.9 資產盤點 / A.5.12 資訊分級 | 強制 owner / data_classification |
| checkov IaC 政策閘門 | P2-4 | A.8.27 安全系統架構 / A.5.36 合規審查 | 政策即程式碼,PR 階段擋下不合規 IaC |
| 可稽核豁免清單(SoA 式) | P2-4 | A.5.36 合規審查 | `.checkov.yaml` 每條 skip 附理由,可逐項稽核 |
| 故意違規 self-test | P2-5 | A.5.36 合規審查 | 證明閘門真的會擋(負向測試) |

### 測試 gate(promote what passed test)

把「測試」做成不可繞過的供應鏈閘門:沒通過測試的程式碼 build 不出可簽章的映像,
測試證據隨 digest 一路過版,部署/過版前重驗。補上「整條鏈沒有測試」的最大缺口。

| 本計畫的技術控制 | 對應 | ISO 27001:2022 控制項 | 說明 |
|------------------|------|----------------------|------|
| paved-road `test` job(mvn test) | `supply-chain-sign.yml` | A.8.29 開發與驗收中的安全測試 | `build-sign` `needs: test`,沒過/空套件就不 build |
| 測試證據隨 digest 過版 | `promote.py` / `validate_promote.py` | A.8.32 變更管理 | `testReport`/`testCount` 與 digest 同搬,防舊證據漂白 |
| 部署/過版前驗測試證據(fail-closed) | `verify_deploy_gate.py` 檢查 4 | A.8.29 安全測試 | 缺證據/空套件/指紋無效 → 拒絕(self-test 證明) |
| 防「綠燈空殼」 | `test` job + 閘門 | A.8.29 安全測試 | 零測試/跳過測試不構成證據,雙層擋 |

| 本計畫的技術控制 | 對應任務 | ISO 20000 領域 | 說明 |
|------------------|----------|----------------|------|
| 標準專案範本 | TASK-02 | 服務目錄 | 範本即活的服務目錄 |
| 預核准範本 | TASK-02 | 標準變更 | 低風險可重複,免逐次審核 |
| PR 流程 | TASK-06 | 變更管理 | 有目的、風險、回退方案 |

## 四、「標準變更」的關鍵論述(對 CAB / 變更委員會)

ISO 20000 本就定義「標準變更」:**預先核准、低風險、可重複的變更,
不需每次走完整審核流程。**

本計畫的「黃金路徑範本」正是標準變更的自動化與規模化:
- 範本內容經一次性嚴格審核(由平台 + 資安群組)
- 之後開發者使用範本 = 套用已核准的標準變更
- 不是繞過變更管理,是實踐其最進步的形態

## 五、落地時必做的「翻譯層」

稽核員不一定看得懂 Git log 與 workflow YAML。因此必須:

1. 把 GitOps 紀錄整理成稽核看得懂的「變更證據」(TASK-07)
2. 每條政策規則對應到 ISO 控制項編號(本表 + 程式碼註解)
3. 證明護欄本身有經過驗證、有測試、有版控

> 這層翻譯能力,就是銀行 IT 在 AI 時代的護城河。
