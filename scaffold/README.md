# scaffold/ — 自助專案範本(黃金路徑 Golden Path)

> **這是什麼**:讓開發者「點一下」就生出一個**預設合規**的新專案,
> 不必自己從零搭 CI、想合規設定、查命名規範。
>
> **治理定位**:本範本即 ISO 20000 的「服務目錄(Service Catalogue)」項目,
> 也是「標準變更(Standard Change)」——預先核准、低風險、可重複,依框架本就
> 不需每次走完整 CAB 審核。詳見 `COMPLIANCE_MAP.md` 第三、四節。

## 這個範本替開發者省掉了什麼

| 開發者原本要自己操心的事 | 用了範本之後 |
|--------------------------|--------------|
| 從零寫 CI workflow | ✅ 內建 `ci.yml`,開箱即跑機敏掃描 + 結構檢查 |
| 研究公司命名 / 結構規範 | ✅ 範本骨架已符合規範,結構檢查自動把關 |
| 擔心金鑰不小心進版控 | ✅ 內建機敏資訊掃描(A.8.12),違規自動擋下 |
| 寫一份合格的 README | ✅ 附 `README.template.md`,填空即可 |
| 設定 `.gitignore` | ✅ 已內建(含 secrets / 常見產出排除) |
| 不知道哪些是「必要文件」 | ✅ 範本已備齊,缺檔會被 CI 提示 |

> **一句話**:開發者只需專注業務邏輯,基礎設施 / 資安 / 合規的複雜性由平台吸收
> (體現「降低認知負荷」與「護欄而非閘門」)。

## 範本內含哪些「預設合規」內容

```
project-template/
├── README.template.md          # README 範本(填空即可)
├── .gitignore                  # 預設排除 secrets 與常見產出
└── .github/workflows/
    └── ci.yml                  # 預設合規 CI:機敏掃描 + 結構檢查骨架
```

每項合規內容對應的 ISO 控制項:

| 範本內含 | 對應 ISO 控制項 | 說明 |
|----------|----------------|------|
| 機敏掃描(`ci.yml`) | ISO 27001 A.8.12 資料外洩防護 | 金鑰不得進版控 |
| 結構 / 命名檢查(`ci.yml`) | ISO 27001 A.5.37 文件化操作程序 | 強制結構一致 |
| 範本本身 | ISO 20000 服務目錄 / 標準變更 | 預核准、可重複 |

## 如何使用(開發者視角)

> ⚠️ 「Template repository」需在 GitHub repo 設定中啟用(見下方「平台維運須知」)。

1. 在 GitHub 上點本 repo 的 **「Use this template」→ Create a new repository**。
2. 新 repo 自動帶有上述 CI 與合規設定,**無需手動設定**。
3. 把 `README.template.md` 改名為 `README.md`,填入專案資訊。
4. 開始開發;每次 PR 會自動跑合規檢查(護欄)。

## 平台維運須知(給平台 + 資安群組)

- **啟用 Template repository**:GitHub → repo **Settings → General →
  Template repository** 打勾。此為一次性設定,無法在本機 git 完成。
- 範本內的合規骨架(`ci.yml`)目前為 **skeleton**;其偵測規則的完整實作與
  「故意違規」測試,於 `[TASK-03]`(機敏掃描)、`[TASK-04]`(結構檢查)補齊。
- 範本內容的變更視同「改護欄」,需平台 + 資安群組審核(見 `CODEOWNERS`)。
