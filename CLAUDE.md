# CLAUDE.md — 給 Claude Code 的專案脈絡

> 此檔放在 repo 根目錄,Claude Code 會自動讀取,作為整個專案的脈絡與執行慣例。

## 專案是什麼

一個銀行內部開發者平台(IDP)的 Phase 1 雛形。核心不是「裝工具」,
而是用最輕的技術(Git + GitHub Actions)體現 Platform Engineering 的三個理念,
並嚴格對齊 ISO 27001 / ISO 20000,產出可稽核的證據。

## 三個必須貫穿所有產出的理念

1. **護欄,而非閘門**:合規寫成程式碼自動執行,不靠人工 review。
2. **平台即產品**:開發者是客戶,在乎採用率與體驗,讓護欄盡量無感。
3. **降低認知負荷**:平台吸收複雜性,開發者只管業務邏輯。

## 執行慣例

- 嚴格按 `PROJECT_PLAN.md` 的 `[TASK-xx]` 順序執行,不跳階段。
- 每個任務完成後停下,讓使用者確認,再進下一個。
- 每寫一條政策規則,都要在程式碼註解標註對應的 ISO 控制項編號
  (參照 `COMPLIANCE_MAP.md`)。
- 涉及機敏資訊處理時,絕不在範例中放入真實憑證,一律用明顯的假值
  (如 `EXAMPLE_FAKE_KEY_DO_NOT_USE`)。
- 政策檔(`policies/`、`.github/workflows/`)的變更,在 PR 描述中
  特別標註「此為護欄變更,需資安審核」(呼應 SoD)。

## 不要做的事

- 不要為了「炫技」引入 Phase 1 不需要的重型工具(K8s 等),除非已通過 Gate 1。
- 不要產出無法對應到治理控制項的規則。
- 不要繞過 PR 直接改 main。

## 完成 Phase 1 的定義

見 `PROJECT_PLAN.md` 第 6 節的 Gate 1 檢查清單,全數打勾才算完成。
