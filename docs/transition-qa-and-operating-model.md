# 長官提問 → 怎麼答:轉移、操作模型、人員組織異動

> **用途**:簡報講完,長官會追問「然後呢——怎麼從頭開始、怎麼慢慢收緊到系統管理、
> 現有管理怎麼轉、人會怎麼動、系統怎麼配合我們要的管理」。這份是**現場 Q&A 的彈藥**:
> 每個預期提問配一個「結論先行」的答法。`[填:…]` 換成公司實況。

---

## 1. 願景:我們要給什麼(一句話 + 三支柱)

> 願景要用**結果**講,不是技術。給管理階層這句:

> **「可證明的合規(provable compliance):任何時刻,我們都能用系統證明——
> 每個系統都套了它該套的控制。不靠人記得、不靠運氣、不靠稽核前補件。」**

三支柱:
1. **控制 by construction**:合規內建在「變更上線的唯一通道」,不是事後檢查。
2. **靠例外與指標治理**:管理層管政策與例外,不逐筆審交易(management by exception)。
3. **合規是平台功能,不是稅**:開發者在鋪好的路上跑,護欄盡量無感。

> 一句電梯話:「我們要把合規從『牆上的政策』變成『系統裡不可繞過的事實』。」

---

## 2. 0→1 的關鍵是什麼

> 長官最常問:「那第一步、最關鍵的是什麼?」

**0→1 = 讓「一條控制」在「一個真實服務」上,端到端 fail-closed 且自動留證,跑起來。**
不是廣度、不是買工具、不是再寫一份政策——是**深度打穿一條,證明這個模式在我們的真系統上成立**。
因為這一條會變成「可複製的範本」,後面每個服務照抄;它也是說服懷疑者的鐵證(會動的東西)。

0→1 的四個必要條件:
1. **一位高層 sponsor + 空中掩護**(擋政治阻力)。 `[填:誰]`
2. **一個願意的試點團隊 + 一個服務**(不需全公司)。 `[填]`
3. **挑一條「痛且看得見」的控制**(贏了大家有感)。 `[填:如機密資料加密 / 部署前驗章]`
4. **產出可複製的鋪好的路範本**(下一個服務 reuse,不重做)。

> 反例(別做):一上來想覆蓋整個 ISO 全控制 → 回到「紙上」;先買大平台 → 工具不等於強制。

---

## 3. 怎麼「漸進收緊」到系統管理(最關鍵的技術答法)

> 長官會擔心:「一下子全擋,不是把大家搞死、把交付卡死?」
> 答案是**一個可調的旋鈕:從監測 → 強制,分五段,不是一夜切換。**(這正是 OPA/Kyverno 的 audit→enforce 模式)

| 段 | 模式 | 做什麼 | 衝擊 |
|---|---|---|---|
| ① 監測 | warn-only | 閘門跑但只記錄/警告,不擋 | 零衝擊,先收集「有多少會 fail」的資料 |
| ② 告知+期限 | warn + deadline | 通知團隊「X 日起會開始擋」,給緩衝去修 | 低,有跑道 |
| ③ 新案強制 | enforce-new | 新變更 fail-closed;既有的先放行(grandfather) | 中,只影響新案 |
| ④ 全面強制+例外出口 | enforce-all | 全部 fail-closed,搭配 time-boxed 例外(waiver) | 高,但有合法出口 |
| ⑤ 收緊基線 | raise-bar | 隨團隊成熟,逐步調高矩陣要求 | 持續演進 |

> 關鍵訊息:**「我們用資料決定何時收緊,不是拍腦袋。先監測看真實衝擊,給跑道,再分批強制,
> 全程留一個合法的例外出口——所以不會把交付卡死,而是可控地收緊。」**

---

## 4. 現有管理系統如何「轉移」(不是打掉重練)

> 長官會問:「那我們現在的 CAB、工單、稽核流程怎麼辦?」
> 答:**Strangler-fig(絞殺榕)漸進包覆——保留、接上、逐步把流量導到新路,不是一次替換。**

| 現有 | 怎麼轉(不廢除,是升級/接上) |
|---|---|
| ITSM / 工單系統 | 保留;把工單號接進閘門(部署要對應一張真實、已核准的工單)。`[填:你們用什麼]` |
| CAB(變更委員會) | **不解散,往上移**:從「審每筆變更」→ 審「標準變更範本(預核准鋪好的路)」+ 治理政策矩陣與例外。開會更少談交易、更多談政策。 |
| 人工變更核准 | 共存期:舊人工流程 + 新自動閘門並行;信心建立後,自動路變預設、人工變例外。 |
| 稽核(年度補件) | 轉「連續保證」:稽核員拿證據平面唯讀權,隨時查,不再一年一次救火。 |

> 關鍵訊息:**「現有流程不廢,是被『接上系統』並升級——CAB 不會失業,他們從蓋章變成定規則。」**
> (這句很重要:消除「我的位子會不會沒了」的抗拒。)

---

## 5. 核心管理機制 + 重點行動(管理階層要 stand up 的東西)

決策端要建立的六個機制(技術只是執行,這些是「管理」本體):
1. **控制基線矩陣**(分級→控制,as-code,治理端擁有)
2. **例外 / 風險接受流程**(time-boxed waiver:owner+到期+理由)← 安全閥,沒它人會偷繞
3. **決策權限模型(RACI)**:誰分級、誰擁矩陣、誰批 waiver、誰在 CAB
4. **治理儀表板**(management by exception:例外數/達標率/漂移/waiver 老化)
5. **定期管理審查**(ISO 27001 §9.3 節奏:控制還合理嗎?例外在堆積嗎?)
6. **當責登記**(每個服務有名字的 risk owner)

**重點行動順序**:① 定 sponsor + RACI → ② 建矩陣 + 上 warn-only 閘門 → ③ 例外流程 + 儀表板 → ④ 審查節奏。

---

## 6. 系統怎麼「配合政策」做到管理階層要的管理

> 長官的潛台詞:「我要的是『我能管、能看、能證明』,系統怎麼給我這個?」

機制 = **policy-as-code 當介面**:
- 管理層用**矩陣(資料)** 表達意圖(什麼分級要什麼控制)——不寫程式。
- 系統用**引擎(閘門/admission)** 強制——缺控制就擋。
- 證據自動流回**儀表板**——管理層看趨勢、裁例外、定期審查。

> 管理層的「管理」動作變成:**調矩陣、裁例外、看儀表板、跑審查**——而不是逐筆檢查。
> 系統回饋給他們的是:**可證明的控制 + 靠例外掌舵 + 連續證據**。這就是他們要的管理,被系統交付。

---

## 7. 人員 / 組織異動方案(誠實,框成「升級」不是「裁減」)

> 長官會問:「那人怎麼動?誰做什麼?誰的位子變了?」
> 鐵則:**0→1 不需要大重組;組織異動跟著『證明』走,不是先動組織。** 這降低政治風險。

### 7.1 角色轉變(是賦權/升級,不是裁員——這樣講)
| 角色 | 從 | 到 |
|---|---|---|
| 資安 / 風險 | 逐筆審查的守門員 | **政策作者 + 例外裁決者**(更高槓桿、少打雜) |
| CAB | 審每筆變更 | 治理政策矩陣 + 標準變更範本 |
| 稽核 | 年度補件救火 | **連續保證**(唯讀證據平面) |
| 開發者 | (不變) | 日常不變,但合規變無感(鋪好的路) |
| (新)平台團隊 | — | 擁有引擎/鋪好的路,當內部產品經營(可由現有 DevOps/infra 升級組成) |
| (新)服務 owner | — | 每個服務有名字的當責人(多半是現有人多戴一頂帽) |

### 7.2 分階段(對齊技術 ramp,不超前)
- **P0–P1 試點**:虛擬團隊,借人兼任,**不正式重組**。sponsor + 1 位平台型工程師 + 1 位資安共筆第一版矩陣。
- **P2–P3 擴張**:成立小型平台團隊、指定治理 owner、點名各服務 owner。
- **P4 規模化**:操作模型全面轉 management by exception;改寫 CAB 章程;稽核拿連續存取。

### 7.3 變更管理(人的那一面,決定成敗)
- **共創而非宣布**:讓「角色會變的人」參與設計新流程(IBM 式嵌入)——參與感降低抗拒。
- **正面處理恐懼**:工作保障、權力、面子。明說「這讓你的角色升級、少打雜」,不是取代你。
- **找 champion**:每個受影響群體找一個擁護者,由他內部帶。
- **技能**:平台工程需 policy-as-code / GitOps 技能(訓練或招 1–2 顆種子);治理端需學「把政策寫成可審查規則」(初期跟平台 pair)。

## 7b. 出事誰負責?— 當責與政治(長官 100% 會問)

> 「每件決策都有政治影響——這麼做,誰引發、誰承擔?」「最終還是要有人負責(像 AI 落地一樣)。」
> 結論先講:**是的,最終一定要有人負責;重點不是『有沒有人負責』,是『當責落在對的人、且講得清楚』。**

### 層層簽核的政治真相:它在「分散當責」
層層簽核表面是控制,骨子裡的政治功能是**分散當責**——大家都簽了 = 法不責眾 = 出事沒有單一個人扛。
這是它**黏著、難砍的真正原因**:它是大家的政治保護傘。代價是當責被稀釋到等於沒有,
出事時要嘛沒人扛、要嘛抓最後執行的網管當替罪羊。

### 誰引發、誰承擔(兩模型對比)
| | 引發(誰造成) | 承擔(誰扛) |
|---|---|---|
| 簽核模型 | **分散**:申請的/批的/執行的互推「我只照單做」 | **模糊** → 常落在最弱的執行者,或找替死鬼。當責是霧 |
| as-code 模型 | **精確不可變**:commit 作者申請、資安批的是這個 diff、閘門強制 or 具名者接受例外 | **落在真正做決策的人**(見下),且對事不對人 |

as-code 模型下「承擔」落在哪(精確且公平):
- **閘門擋了該擋的、卻仍出事** → 是「政策(矩陣)錯了」→ 當責在**政策擁有者**,而且這是「改進事件」不是「抓戰犯」;照規則跑的工程師**無責**。
- **有人批了例外/風險接受才出事** → 那個**具名、簽了時間戳理由**的人扛。公平,因為他自願接受了風險。
- **沒走流程的變更** → 漂移偵測**指名道姓**是誰、何時偏離。

### AI 落地的同型問題
AI 落地不會「怪工具」,而是要有**具名的人對『用它的決策』負責**。治理一樣——
**系統不是用來取代負責的人,是用來讓『誰負責』從一團霧變成白紙黑字,且保護只是照規則跑的基層。**

### 對「推這件事的人」的政治(誰引發誰承擔的另一層)
推改革本身就是政治行為。**別當那個引發改革、卻獨自承擔反彈的小員工。**
- 讓高層 **sponsor 當『引發者』**(決策是他的、功勞是他的),讓**系統當承擔機制**(對事不對人、有證據),你當**賦能者**。
- 這個模型**保護基層**(網管/工程師從替罪羊變「我照閘門跑、無責」)→ 跟最容易被當替死鬼的那群人**結盟**,他們挺你。
- 真正會抵抗的是**靠分散當責取得保護傘的中間簽核層**——他們失去的是「霧」。靠 sponsor 空中掩護 + 把他們角色重新定位成「政策作者/真判斷」化解。

> 一句話:**簽核把當責變成霧(法不責眾);as-code 把當責變成白紙黑字(政策擁有者、具名例外者),
> 而且保護照規則跑的人。最終都要有人負責——差別是『負責的人是不是對的、講不講得清楚』。**

## 8. 預期提問 → 一句話答(進房前掃一遍)

| 長官會問 | 你的一句話答 |
|---|---|
| 怎麼從頭開始? | 「一條控制、一個服務、端到端跑起來——0→1 是深度打穿一條,不是廣度。」 |
| 怎麼漸進不卡死交付? | 「一個可調旋鈕:監測→告知→新案強制→全面強制+例外出口。用資料決定何時收緊。」 |
| 現有 CAB/工單/稽核怎麼辦? | 「不打掉,接上並升級:CAB 從蓋章變定規則,稽核從年度補件變連續保證。」 |
| 核心要做什麼? | 「六個管理機制:矩陣、例外流程、權限 RACI、儀表板、定期審查、當責登記。」 |
| 系統怎麼配合我要的管理? | 「policy-as-code 當介面:你調矩陣、裁例外、看儀表板;系統強制並回證據。」 |
| 人怎麼動? | 「0→1 不重組,借人兼任;組織跟著證明走。資安從守門員升級成政策作者。」 |
| **出事誰負責?** | 「簽核把當責變成霧(法不責眾);as-code 把它變白紙黑字——閘門擋了還出事是政策擁有者(改進非抓人)、批例外才出事是具名者扛、沒走流程漂移偵測指名。照規則跑的基層無責。」 |
| 願景一句話? | 「把合規從牆上的政策,變成系統裡不可繞過的事實。」 |

> 全程定位:你是「把長官的願景落地、可運轉」的人。給路線、給證據,讓長官當決策的英雄。
