# 從「N 張申請單」到「一個服務請求 + 自動控制」

> **公司現況**:部署一個服務 = 硬體一張申請單、middleware 一張、網路 N 張、
> 白箱掃描一張、黑箱掃描一張、環境弱點掃描一張……每張公文式層層簽核。
> **問題**:怎麼解套?——先把問題**定義對**,解套就浮出來了。

---

## 1. 問題定義(定義對了,解法自明)

這不是「表單太多」的效率問題,而是**兩個範疇被混為一談**的結構問題。

### 病灶:把「簽核」當「控制」,把「申請單」當「真相」

一張「黑箱掃描申請單」被簽核了,證明的是 **「有人核准了『要掃描』這件事」**,
**不是**「掃描跑過了、沒有高風險漏洞」。
> **簽名證明『意圖 / 授權』,不證明『執行 / 結果』。**

你們蓋了一堆章,卻仍**無法證明系統是安全的**——因為簽核回答的是「准不准做」,
不是「做了沒、做對沒」。這是整個問題的核心範疇錯誤。

### 三個結構病

1. **流程碎片化**:一件邏輯上的事(上線一個服務)被打散成 N 條**互不相連**的簽核流;
   沒有一條線把它們串成「這個服務的真實狀態」。要回答「全部做了嗎」得人工收集 N 張單。
2. **簽核 ≠ 控制**:確定性的技術檢查(掃描過沒過、組態合不合規、網路規則在不在政策內)
   被當成「公文」用人簽核——但這些是機器能判定的事實,簽名不會讓它更真。
3. **認知負荷外推**:複雜性被丟給「人」——申請者得知道要填哪 N 張、什麼順序、各走哪條簽核鏈。
   平台沒有吸收複雜性,反而把它轉嫁給使用者。

### 一句話診斷(給長官)

> **「我們在用『公文流程』管『技術控制』。但技術控制是確定性的——該用閘門『跑』出結果,
> 不該用簽名『核』出意圖。簽一堆章,證明的是有人核准,不是系統安全。」**

---

## 2. 解套:分開兩件事,各綁對的機制

關鍵動作=**把『意圖/授權』和『執行/證據』拆開**,各自綁到對的東西:

| | 本質 | 該用什麼 | 結果 |
|---|---|---|---|
| **意圖 / 授權** | 「我們要做這件事,有權的人同意」 | **一張**服務請求 + (必要時)人簽核 | 收斂成 1 張,不是 N 張 |
| **執行 / 證據** | 「控制真的跑了、過了」 | **自動閘門**,產出 pass/fail 物證 | 掃描/組態/網路規則變閘門,不是表單 |

### 2.1 N 張申請單 → 一張服務請求 + 平台自動展開
申請者只填 **一張**「我要服務 X、環境 Y、資料分級 Z」。
硬體、middleware、網路、各種掃描變成那張請求的**衍生需求**,由平台自動展開,
**不是申請者要分別去填的 N 張單**。(這就是「平台吸收複雜性、降低認知負荷」。)

### 2.2 確定性的檢查 → 閘門,不是申請單
掃描類尤其是最明顯的範疇錯誤——**你不會「申請去做掃描」,掃描就是個會跑、會擋、會留證的閘門。**

| 現在的申請單 | 變成什麼 |
|---|---|
| 硬體申請 | 服務請求裡宣告資源規格 → IaC 佈建 + 政策閘門驗(資源即程式碼) |
| middleware 申請 | 宣告 + IaC + 組態政策閘門(itops 的 CKV_OL_1~6 已示範) |
| 網路 N 張申請 | 網路政策即程式碼;N 張收斂成「宣告的網路需求」+ 政策閘門驗 |
| 白箱掃描(SAST)申請 | pipeline 自動閘門,產判定物證(無表單) |
| 黑箱掃描(DAST)申請 | pipeline 自動閘門,產判定物證(無表單) |
| 環境弱點掃描申請 | 自動閘門,產判定物證(無表單) |

### 2.3 簽核只留給「真的需要人判斷」的地方
能機器判定的(掃描過沒過、組態合不合規)→ **閘門**,不留簽核。
只有**真正需要人類判斷**的才保留人簽核:**風險接受 / 例外 / 正式區 CAB 核准**。
> 結果:層層簽核從「每件事都簽」收斂成「只簽真正的決策」。

---

## 3. 為什麼「層層簽核」會存在 — 以及為什麼閘門一上它就該瘦身

**層層簽核多半是「補償行為」**:因為系統不強制控制,只好用人「層層把關」當唯一的控制。
每一層簽核,其實是在補一個「系統沒擋」的洞。

> **一旦閘門強制執行,多數簽核層就失去存在理由**——它們本來就是在替代不存在的自動化。

辨識每一層簽核:
- **加值的簽核**(真的有人在評估風險、做判斷)→ 保留,但串進系統(綁進 PR/工單)。
- **橡皮圖章**(因為流程規定才簽、不加判斷)→ 閘門上線後可移除,它的工作閘門做得更好更快。

---

## 4. 漸進解套(不大爆炸,對齊 strangler-fig + 監測→強制旋鈕)

不要 day 1 廢掉所有申請單(會炸鍋)。順序:
1. **挑最確定性的先轉**:三種掃描申請單最該先死——它們是純確定性檢查,變成 pipeline 閘門,
   閘門過了**自動填**那張單的證據、自動核准;人不用再簽。
2. **單張內容先自動驗**:某張單的內容先讓系統自動檢查(warn-only),再進到「閘門過才放行」。
3. **N 張逐步收斂成 1 張**:平台把衍生控制一個個吸收進來,申請者要填的單越來越少。
4. **簽核層逐步瘦身**:每強制一道閘門,就檢視「哪層簽核變成橡皮圖章」→ 移除。
5. **保留真判斷的出口**:風險接受/例外/CAB 用 time-boxed waiver,合法地留給人。

> 這條完全套用 [`transition-qa-and-operating-model.md`](transition-qa-and-operating-model.md) 的五段旋鈕與 strangler-fig。

## 5. itops 已經示範了這個收斂(可現場 demo)

itops 的黃金路徑就是這題的活證明:
**一張服務請求(Issue Form)→ PR → 供應鏈自動跑 SAST/DAST/SBOM/掃描閘門 → 部署前驗章 → CMDB**。
N 張掃描申請單在這裡是**自動閘門 + 自動物證**,不是表單;申請者只開**一張**請求。

> 對長官的話:**「不是要你們廢掉治理,是把『確定性的檢查』從公文簽核變成會跑的閘門,
> 把 N 張申請單收斂成一張服務請求。治理更嚴(系統強制)、但更快(不靠層層蓋章)、且可證明。」**

## 5b. worked example:防火牆 —「層層簽核是為了追溯」的真相

> **角色**:申請=程式人員、審核=資安、執行=**網管手動改防火牆**。
> **挑戰**:「防火牆審核機制出問題了怎麼追溯?層層簽核不就是為了追溯嗎?」

**先承認對的部分**:對,層層簽核確實是為了追溯——**但它只追溯到「決策」。**
它有兩個致命洞,而防火牆出事**正好出在這兩個洞**:

1. **追溯不到「執行」**:簽核證明「資安批准了開 port X」,但網管是**手動**套規則。
   live 防火牆上實際長怎樣,跟「批准的內容」之間**沒有系統綁定**——
   網管可能打錯、多開來源、開錯網段。你能追到「誰批准」,**追不到「實際改的 == 批准的」**。
2. **看不到「未授權變更」**:沒走單的 out-of-band 規則(網管臨時手動加的、或舊規則沒清)
   **完全在簽核紀錄之外**——直到滲透測試或事故才被發現。

> 同一個範疇錯誤再現:**簽核 = 決策意圖的追溯,不 = 執行真相的追溯。**

### 防火牆出事,兩種模型怎麼追溯(並列)

| 追溯問題 | 現況(申請單+手動執行) | 解套後(規則 as code + 自動套用 + 漂移偵測) |
|---|---|---|
| 誰申請 / 誰批准 / 為何 | 簽核單(若單還在) | PR:commit author=申請者、資安 review=核准、justification+工單=為何 |
| **實際改了什麼** | 自由文字描述,**未必等於 live 規則** | PR 的**精確 diff**=資安批的就是實際規則文字 |
| **執行的 == 批准的?** | **無法保證**(手動套,人會錯) | **by construction 相等**:reconciler 套的就是那個 commit |
| 誰執行的 / 何時 | 網管手動,常共用帳號、無逐筆歸屬 | apply log:哪個 commit、何時、由誰/由 reconciler 套 |
| **沒走流程的規則** | **看不到**,事故才發現 | **漂移偵測**:live ≠ git → 即時告警 / 自動回退 |

### 解套:防火牆規則 as code → 追溯反而更強(三層全可證)
- **決策追溯**:每筆規則變更走 PR。申請=PR 作者、核准=資安在 PR 上 review(codeowner)、
  為何=PR 說明+工單、**改了什麼=精確 diff**。比表單強:資安批的是**實際規則文字**,不是描述。
- **執行追溯**:GitOps reconciler 自動套用 → **live 狀態 == 批准的 commit,by construction**;
  apply log 留「哪個 commit、何時」。網管不再手動打字 = 消除最弱環。
- **未授權變更追溯**:**漂移偵測**連續比對 live 防火牆 vs git;任何沒有 approved source 的規則
  = 漂移 = 告警/自動回退。**這是層層簽核永遠給不了的能力。**

### SoD 不但保留,還更強
申請(dev 開 PR)≠ 核准(資安 review)≠ 執行(自動套用 / 網管引用 commit)——
系統強制 requester≠approver,且執行自動化 → **沒有人能又批又默默改**。三權分立由系統落實,不靠流程自律。

### 過渡(網管手動這個最弱環怎麼漸進)
- 理想:防火牆 as code + 自動套用,網管退出手動。
- 過渡(還沒自動套用):至少**手動變更要引用 approved commit** + 上**漂移偵測**——
  手動偏離批准內容、或 out-of-band 規則,立刻被抓。先補「可偵測」,再進「自動套用」。

> **殺手句**:**「層層簽核給你『決策的追溯』,給不了『執行的追溯』和『未授權變更的偵測』。
> 防火牆出事,十之八九出在後兩者——網管打錯/多開,或根本沒走單的規則。
> 把規則變成 git + 自動套用 + 漂移偵測,這三層追溯才都成立,而且比簽核更強、更即時。」**

## 6. 一頁總結

| | 現在(申請單治理) | 解套後(服務請求 + 閘門) |
|---|---|---|
| 入口 | N 張申請單,各自簽核 | 1 張服務請求 |
| 控制 | 人簽核(證明意圖) | 自動閘門(證明結果)+ 留物證 |
| 掃描 | 一張張申請單 | pipeline 閘門,fail-closed |
| 簽核 | 層層蓋章(多為補償) | 只留真判斷(風險/例外/CAB) |
| 真相 | 散在 N 張單,要人工收集 | 單一服務請求 + CMDB,系統可查 |
| 能證明安全嗎 | 否(只證明有人簽) | 是(閘門+物證隨時可證) |

> 一句話:**把「一個服務 N 張申請單、層層簽核」收斂成「一張服務請求 + 平台自動展開的控制」;
> 能機器判定的變成會出證據的閘門,只有真正要人判斷的才留簽核。**
